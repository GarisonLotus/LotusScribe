## project.yml essentials

- `name: LotusScribe`; deployment target macOS 14.0 (D5).
- App target `LotusScribe` (type `application`, platform macOS):
  - `PRODUCT_BUNDLE_IDENTIFIER: com.garisonlotus.LotusScribe` (D2)
  - Ad-hoc signing (D6): `CODE_SIGN_IDENTITY: "-"`, `CODE_SIGN_STYLE: Manual`
  - Generated Info.plist via XcodeGen `info:` block with `LSUIElement: true`
  - No dependencies (D7 — no third-party packages anywhere in Phase 0)
- Test target `LotusScribeTests` (type `bundle.unit-test`, platform macOS):
  - Depends on `LotusScribe`; XcodeGen auto-sets `TEST_HOST` (hosted tests — required
    to `@testable import LotusScribe` from an app target)
  - `GENERATE_INFOPLIST_FILE: YES` — required workaround (R2): xcodegen emits no
    Info.plist for unit-test bundles, and code signing needs one. Do not drop in 0B+.
- Scheme `LotusScribe` with `LotusScribeTests` in its test action (needed for `xcodebuild test`).

## Makefile recipe (engineer implements)

- `make generate` → `xcodegen generate`
- `make build` → `xcodebuild -project LotusScribe.xcodeproj -scheme LotusScribe -configuration Debug build`
- `make test` → `xcodebuild test -project LotusScribe.xcodeproj -scheme LotusScribe -destination 'platform=macOS'`

`build` and `test` should depend on `generate`. No other recipes in Phase 0.

## Sub-phase 0A — Scaffold + menu-bar presence

Deviation from suggested slicing: the test bundle target, scheme test action, and one
smoke test land in 0A (not 0B), so `make test` is green from the first commit and
project.yml does not churn in 0B. Otherwise as suggested.

**Deliverables (LoC budgets are ceilings, not targets):**
- `project.yml` (~40), `Makefile` (~20), `.gitignore` additions (~5)
- `main.swift` (~10), `AppDelegate.swift` (~20), `StatusItemController.swift` (~40):
  status item with a fixed SF Symbol or title, menu containing a single Quit item.
- `Tests/LotusScribeTests/SmokeTests.swift` (~10): one trivial Swift Testing `@Test` (D4).

**Verify:**
1. `make generate && make build && make test` all succeed.
2. Launch the built .app: status item appears in the menu bar; Quit item terminates it.
3. While running: no Dock icon; app absent from Cmd-Tab switcher.
4. `git status` shows no `.xcodeproj` or DerivedData tracked.

**Invariants established:**
- Menu-bar-only: `LSUIElement = YES`; no Dock icon, no Cmd-Tab entry, ever.
- `.xcodeproj` is generated, never committed (D1).
- App target carries zero third-party dependencies.

## Sub-phase 0B — Settings storage + tests

**Deliverables:**
- `SettingsStore.swift` (~50): UserDefaults-backed, injectable for tests. Illustrative shape:

  ```swift
  final class SettingsStore {
      init(defaults: UserDefaults = .standard)
      var sttEndpointURL: String? // key "sttEndpointURL"
      var sttModel: String?       // key "sttModel"
      var llmEndpointURL: String? // key "llmEndpointURL"
      var llmModel: String?       // key "llmModel"
  }
  ```

  Keys are exactly the four above (PLAN.md Architecture); all default nil. No
  registration of defaults, no observation, no other keys — later phases add them.
  No settings UI (D3).
- `KeychainStore.swift` (~70): own wrapper over Security framework generic passwords (D7).

  ```swift
  struct KeychainStore {
      init(service: String = "com.garisonlotus.LotusScribe")
      func set(_ secret: String, for account: String) throws
      func get(_ account: String) throws -> String?
      func delete(_ account: String) throws
  }
  ```

  API keys are per-endpoint: accounts `"stt-api-key"` and `"llm-api-key"`.
- `Tests/LotusScribeTests/SettingsStoreTests.swift` (~50): round-trip each key;
  persistence across store instances sharing one `UserDefaults(suiteName:)`;
  suite removed in teardown.
- `Tests/LotusScribeTests/KeychainStoreTests.swift` (~60): set/get/overwrite/delete
  round-trips; `get` of a missing account returns nil. Tests use a distinct service
  name (e.g. `com.garisonlotus.LotusScribe.tests`) and delete their items in teardown.

**Verify:**
1. `make test` green (smoke + both new suites).
2. Persistence across relaunch, storage-layer level (no UI exists — D3):
   write via a small test or `defaults write com.garisonlotus.LotusScribe sttModel whisper-1`,
   relaunch the app, confirm `defaults read com.garisonlotus.LotusScribe sttModel` returns it
   and a fresh `SettingsStore` reads it back.
3. After a Keychain test run: `security find-generic-password -s com.garisonlotus.LotusScribe.tests`
   finds nothing (teardown cleaned up).

**Invariants established:**
- Settings live in standard UserDefaults under the four keys above; they survive relaunch.
- Keychain items are scoped to service `com.garisonlotus.LotusScribe`; secrets never
  touch UserDefaults.
- Tests never write to `.standard` defaults or the app's real Keychain service.
