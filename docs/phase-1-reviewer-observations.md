| R8 | 1B | **BLOCKER — data race on `AudioRecorder.converter`.** `private var converter: AVAudioConverter?` is written on the main thread (`start()` assigns, `stop()` nils) and read on the audio render thread (`appendConverted`'s `guard let converter`), with no synchronization — the NSLock guards only `pcm`. `removeTap(onBus:)` does not synchronize with an in-flight tap callback, so `stop()`'s `converter = nil` (ARC release) can race the audio thread's load+retain → over-release/use-after-free. TSan-visible; rare crash in production. Surgical fix: capture the freshly created converter in the tap closure (`installTap { [weak self] buffer, _ in self?.appendConverted(buffer, using: converter) }`) and delete the `converter` stored property — no shared mutable state remains. The `pcm` path itself is correctly locked (post-stop stragglers are cleared by `pcm.removeAll()` at next start). **Fix applied and re-reviewed:** stored property deleted, converter is a local `let` captured immutably by the tap closure, `appendConverted(_:using:)` takes it as a parameter; stop()/catch no longer touch converter state, tap-removal-on-throw intact. Reviewer re-ran `make test`: 32/4 green. Race eliminated, no behavior change beyond the fix | resolved (1B) |
| R9 | 1B | Converter tail never flushed: `stop()` discards frames buffered inside AVAudioConverter (resampler filter delay, ~ms). Inaudible for dictation (user releases after speaking); a final `.endOfStream` drain would add complexity for no 1B value. No action | accepted |
| R10 | 1B | Temp WAVs (`LotusScribe-<UUID>.wav`) are written but never deleted — they accumulate in `temporaryDirectory` until OS purge. Spec §1B mandates the temp-file hand-off itself and 1C replaces the write with TranscriptionService, so this self-resolves; flagging so 1C review confirms the write path is actually removed. **1C review confirmed:** the write path is deleted from `DictationController.stopRecording()` (replaced by TranscriptionService call, not kept alongside); grep of `Sources/` for `FileManager` / `write(to` / `temporaryDirectory` / `.wav` shows zero file-write residue | resolved (1C) |
| R11 | 1C | In-flight transcription is not cancelled or coalesced on a new dictation: `stopRecording()` spawns an unstructured `Task` per key-up, so a user who starts a second dictation while the first transcribe is still in flight gets two concurrent requests completing in arbitrary order. Harmless in 1C (each just logs its transcript) and no retain issue (task captures the value-type service, not `self`). But 1D wires the same completion to *insertion* — a slow/stale transcript could paste after a newer one or into a different focused app. 1D must define ordering/cancellation (e.g. cancel or drop the stale task on the next `startRecording`) | open (1D must address) |
| R12 | 1C | The "20 s timeout enforced" invariant (spec §1C) is set in code (`URLRequest(url:timeoutInterval: 20)`) but not guarded by any test — a one-line `#expect(request.timeoutInterval == 20)` in `requestMatchesSpec()` would make the invariant regression-proof (URLProtocol preserves `timeoutInterval` on the captured request). Minor; code verified correct by inspection | open (cheap follow-up) |
| R13 | 1C | LoC: TranscriptionServiceTests.swift is 144 code lines vs the spec's ~90 estimate. Overage is the D19 `StubURLProtocol` + `bodyData(of:)` stream-drain infrastructure (~40 lines) the spec estimate under-scoped, not test-logic sprawl — R6 precedent applies. Accepted | accepted |
| R14 | 1C | `MultipartBody.data` is a non-mutating computed property (parts + fresh closing boundary each read), so parts can be appended after a read and every read is a complete valid body. Not a footgun: no finalize-then-mutate corruption is possible, and the only production caller builds and reads once inside `transcribe(wav:)`. No action | accepted |
